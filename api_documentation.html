API & Protokoll-Dokumentation

AI Development Tool Chain · Agenten-Kommunikation, Protokolle & Schnittstellen · 2026-06-18

Slash Commands Handoff-Protokoll Gate-Protokoll Artefakt-API Phase-State-API Git Hooks
Über diese Dokumentation Die Tool Chain ist eine agentische Architektur ohne klassische REST-APIs. Die "APIs" sind Kommunikationsprotokolle zwischen Agenten: Slash Commands als Aktivierungsschnittstelle, strukturierte Artefakt-Formate als Datenaustausch, und das Handoff-Protokoll als Inter-Agenten-Übergabe.
1

Slash Command Architektur

Commands sind Markdown-Dateien unter .claude/commands/. Claude Code liest und interpretiert sie als Systemanweisungen für den jeweiligen Agenten. Jeder Command entspricht einer Agenten-Aktivierung.

Handoff-Protokoll (Inter-Agenten-Übergabe)

Jede Agenten-Session endet mit einem standardisierten Handoff-Block am Ende des Artefakts. Der empfangende Agent liest diesen Block als ersten Schritt seiner Aktivierung.

Datum: 2026-06-18
Von: BA | An: AR
Nächster Command: /architect
Artefakte: REQ-001, US-001..US-012

Gate-Protokoll (Qualitätssicherung)

Vor jedem Phasenübergang prüft der Orchestrator definierte Gate-Kriterien. Schweregrade bestimmen ob der Workflow blockiert, warnt oder weiterläuft.

CHECK->>CHECK: status-field == "APPROVED"? CHECK->>CHECK: count(US-NNN) >= 3? CHECK->>CHECK: cross-ref ADR-001? alt BLOCKER CHECK-->>ORCH: ❌ BLOCKED: REQ nicht APPROVED ORCH->>LOG: Protokolliere Gate-Failure else APPROVED CHECK-->>ORCH: ✅ PASS — alle 4 Kriterien erfüllt ORCH->>LOG: Protokolliere Gate-Pass ORCH->>ORCH: Aktiviere nächsten Agenten end
Prüfmethode Beschreibung Beispiel
file-exists Datei muss vorhanden sein REQ-001.md exists
status-field YAML-Frontmatter Status-Wert status: APPROVED
count-min Mindestanzahl Artefakte US-*.md count >= 3
cross-ref Verweis auf anderes Artefakt ADR-001 referenced in REQ
command-exit-0 Shell-Command erfolgreich npm test exits 0
self-assertion Agent erklärt Kriterium erfüllt All acceptance criteria defined

Phase State API (.phase-Datei)

Jedes Projekt hat eine .phase-Datei als YAML. Sie ist das "Single Source of Truth" für den aktuellen Projektzustand — wird vom Orchestrator geschrieben und gelesen.

Git Hook Schnittstellen

Git Hooks automatisieren Qualitätssicherung direkt im Commit-Flow. Pre-commit blockiert bei Verletzungen, post-commit warnt nur.

sequenceDiagram participant DEV as Entwickler participant GIT as Git participant PRE as pre-commit Hook participant POST as post-commit Hook participant CI as CI/CD (optional) DEV->>GIT: git commit -m "feat: ..." GIT->>PRE: trigger pre-commit PRE->>PRE: 1. Lint prüfen (.toolchain-config) PRE->>PRE: 2. Datei-Header vorhanden? PRE->>PRE: 3. Secret-Scan (regex) PRE->>PRE: 4. TODO-Format korrekt? alt Verletzung gefunden PRE-->>GIT: exit 1 (BLOCKIERT) GIT-->>DEV: ❌ Commit abgelehnt + Fehlermeldung else Alles OK PRE-->>GIT: exit 0 GIT->>GIT: Commit durchgeführt GIT->>POST: trigger post-commit POST->>POST: INDEX.md aktuell? alt INDEX veraltet POST-->>DEV: ⚠️ WARNUNG: INDEX.md updaten end GIT->>CI: push → CI Pipeline end
pre-commit: Blockierend — verhindert Commit bei Verletzung
post-commit: Warnend — Commit bereits durch, nur Hinweis
CI/CD: Optional, projektspezifisch konfiguriert
Hook Prüfung Bei Verletzung Konfiguration
pre-commit Lint-Tool (aus .toolchain-config) exit 1 (Commit blockiert) .toolchain-config.lint_command
pre-commit Datei-Header (erste Zeile Kommentar) exit 1 mit Dateiliste Hartcodiert in Hook
pre-commit Secret-Scan (password, api_key, token…) exit 1 mit Fund-Info Regex-Liste im Hook
pre-commit TODO-Format // TODO(KÜRZEL): ... exit 1 mit Beispiel Erlaubte Kürzel im Hook
post-commit INDEX.md in geänderten Verzeichnissen Warnung (exit 0) Hardcoded Ausnahmeliste
